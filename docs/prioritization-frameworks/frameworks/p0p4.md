# P0-P4 Priority Hierarchy Framework

## Overview

The P0-P4 Priority Hierarchy is a severity-based prioritization system commonly used in incident management, production environments, and agile development teams. It provides a clear, universally understood language for communicating task urgency and business impact.

## Framework Details

### Priority Levels

| Level | Name | Response Time | Description | Examples |
|-------|------|---------------|-------------|----------|
| **P0** | Critical | Immediate (< 15 min) | System down, data loss, security breach | Production outage, security vulnerability, data corruption |
| **P1** | High | Within 1 hour | Major functionality broken, significant user impact | Feature outage, performance degradation, payment failures |
| **P2** | Medium | Within 24 hours | Moderate impact, workaround exists | Bug in non-core feature, UI issues, minor performance impact |
| **P3** | Low | Within 1 week | Minor issue, quality improvement | Cosmetic bugs, documentation updates, nice-to-have features |
| **P4** | Trivial | Backlog | Minimal impact, future consideration | Enhancement requests, polish, tech debt (non-critical) |

## Scoring Guidelines

### Priority Score Calculation

```typescript
interface SeverityFactors {
  userImpact: number;      // Percentage of users affected (0-100)
  dataRisk: number;        // 1-10 severity
  revenueImpact: number;   // $ per hour or percentage
  securityRisk: number;    // 1-10 severity
  reputationRisk: number;  // 1-10 severity
}

function calculatePriorityScore(factors: SeverityFactors): number {
  const weights = {
    userImpact: 0.25,
    dataRisk: 0.25,
    revenueImpact: 0.20,
    securityRisk: 0.20,
    reputationRisk: 0.10
  };

  const normalizedUserImpact = factors.userImpact / 10; // Convert to 1-10

  const weightedScore =
    normalizedUserImpact * weights.userImpact +
    factors.dataRisk * weights.dataRisk +
    factors.revenueImpact * weights.revenueImpact +
    factors.securityRisk * weights.securityRisk +
    factors.reputationRisk * weights.reputationRisk;

  // Convert to 0-100 scale
  return Math.round(weightedScore * 10);
}

function determinePriorityLevel(score: number): PriorityLevel {
  if (score >= 90) return 'P0';  // Critical
  if (score >= 70) return 'P1';  // High
  if (score >= 50) return 'P2';  // Medium
  if (score >= 25) return 'P3';  // Low
  return 'P4';                    // Trivial
}
```

### Factor Assessment Guidelines

#### User Impact

| Score | Description | Criteria |
|-------|-------------|----------|
| 10 | 100% users | All users affected, complete outage |
| 8-9 | >50% users | Majority of users affected |
| 6-7 | 20-50% users | Significant portion affected |
| 4-5 | 5-20% users | Moderate user impact |
| 2-3 | 1-5% users | Small group affected |
| 1 | <1% users | Edge case, rare conditions |

#### Data Risk

| Score | Description | Criteria |
|-------|-------------|----------|
| 10 | Data loss | Permanent data loss or corruption |
| 8-9 | Data integrity | Potential data corruption, recoverable |
| 6-7 | Data exposure | Sensitive data potentially exposed |
| 4-5 | Minor data issues | Temporary data inconsistency |
| 2-3 | No data risk | Data unaffected |

#### Revenue Impact

| Score | Description | Criteria |
|-------|-------------|----------|
| 10 | Critical | >$100K/hour or business-threatening |
| 8-9 | Severe | $50K-100K/hour impact |
| 6-7 | High | $10K-50K/hour impact |
| 4-5 | Moderate | $1K-10K/hour impact |
| 2-3 | Low | <$1K/hour impact |
| 1 | None | No direct revenue impact |

## Auto-Fill Logic

### Complete Auto-Fill Engine

```typescript
class P0P4AutoFillEngine {
  constructor(private context: P0P4Context) {}

  async suggestPriority(item: BacklogItem): Promise<P0P4Result> {
    const severityFactors = this.analyzeSeverityFactors(item);
    const priorityScore = this.calculateScore(severityFactors);
    const priorityLevel = this.determineLevel(priorityScore);

    return {
      itemId: item.id,
      priorityLevel,
      priorityScore,
      normalizedScore: priorityScore, // Already 0-100
      severityFactors,
      recommendedResponse: this.getResponseGuidance(priorityLevel),
      escalationPath: this.getEscalationPath(priorityLevel),
      slaDeadline: this.calculateSLADeadline(priorityLevel),
      rationale: this.generateRationale(item, priorityLevel, severityFactors),
      autoFillUsed: true
    };
  }

  private analyzeSeverityFactors(item: BacklogItem): SeverityFactors {
    const text = `${item.title} ${item.description}`.toLowerCase();

    // User Impact Analysis
    let userImpact = 1;
    if (/all users|system down|outage|complete/i.test(text)) userImpact = 10;
    else if (/majority|most users|>50%|widespread/i.test(text)) userImpact = 8;
    else if (/significant|many users|20/i.test(text)) userImpact = 6;
    else if (/some users|minority|10/i.test(text)) userImpact = 4;
    else if (/few users|edge case|rare/i.test(text)) userImpact = 2;

    // Data Risk Analysis
    let dataRisk = 1;
    if (/data loss|data corruption|permanent|deleted/i.test(text)) dataRisk = 10;
    else if (/data integrity|potential loss|recoverable/i.test(text)) dataRisk = 8;
    else if (/data exposure|sensitive|PII|leak/i.test(text)) dataRisk = 7;
    else if (/inconsistency|sync issue|temporary/i.test(text)) dataRisk = 4;

    // Revenue Impact Analysis
    let revenueImpact = 1;
    if (/revenue loss|payment failed|transaction/critical/i.test(text)) {
      revenueImpact = 9;
    }
    if (/checkout|purchase|subscription|billing/i.test(text)) {
      revenueImpact = Math.max(revenueImpact, 7);
    }
    if (/internal|admin|back office/i.test(text)) {
      revenueImpact = Math.min(revenueImpact, 4);
    }

    // Security Risk Analysis
    let securityRisk = 1;
    if (/security|vulnerability|breach|attack|exploit|CVE/i.test(text)) {
      securityRisk = 10;
    }
    if (/authentication|authorization|access control/i.test(text)) {
      securityRisk = Math.max(securityRisk, 7);
    }
    if (/password|credential|token/i.test(text)) {
      securityRisk = Math.max(securityRisk, 6);
    }

    // Reputation Risk Analysis
    let reputationRisk = 1;
    if (/public|media|viral|twitter|social/i.test(text)) reputationRisk = 9;
    if (/customer complaint|churn|negative/i.test(text)) reputationRisk = 7;
    if (/brand|image|trust/i.test(text)) reputationRisk = 6;

    // Override based on explicit priority hints
    if (/p0|critical|sev1|severity 1/i.test(text)) {
      return { userImpact: 10, dataRisk: 10, revenueImpact: 10, securityRisk: 10, reputationRisk: 9 };
    }
    if (/p1|high|sev2|severity 2/i.test(text)) {
      return { userImpact: 8, dataRisk: 7, revenueImpact: 8, securityRisk: 5, reputationRisk: 7 };
    }

    return { userImpact, dataRisk, revenueImpact, securityRisk, reputationRisk };
  }

  private calculateScore(factors: SeverityFactors): number {
    return calculatePriorityScore(factors);
  }

  private determineLevel(score: number): PriorityLevel {
    return determinePriorityLevel(score);
  }

  private getResponseGuidance(level: PriorityLevel): string {
    const guidance = {
      P0: 'IMMEDIATE: Page on-call, assemble incident response team, start war room',
      P1: 'URGENT: Page on-call, notify team lead, begin investigation within 1 hour',
      P2: 'MODERATE: Assign to team member, investigate within 24 hours',
      P3: 'LOW: Add to sprint backlog, address within the week',
      P4: 'MINIMAL: Add to backlog, address when capacity allows'
    };
    return guidance[level];
  }

  private getEscalationPath(level: PriorityLevel): string[] {
    if (level === 'P0') {
      return ['On-call Engineer', 'Team Lead', 'Engineering Manager', 'VP Engineering', 'CEO'];
    }
    if (level === 'P1') {
      return ['On-call Engineer', 'Team Lead', 'Engineering Manager'];
    }
    if (level === 'P2') {
      return ['Assigned Engineer', 'Team Lead'];
    }
    return ['Assigned Engineer'];
  }

  private calculateSLADeadline(level: PriorityLevel): Date {
    const now = new Date();
    const responseTimes = {
      P0: 15 * 60 * 1000,    // 15 minutes
      P1: 60 * 60 * 1000,    // 1 hour
      P2: 24 * 60 * 60 * 1000, // 24 hours
      P3: 7 * 24 * 60 * 60 * 1000, // 7 days
      P4: 30 * 24 * 60 * 60 * 1000  // 30 days
    };
    return new Date(now.getTime() + responseTimes[level]);
  }

  private generateRationale(
    item: BacklogItem,
    level: PriorityLevel,
    factors: SeverityFactors
  ): string {
    const factorsStr = Object.entries(factors)
      .filter(([_, v]) => v >= 7)
      .map(([k, v]) => `${k}: ${v}/10`)
      .join(', ');

    return `Classified as ${level} (${PRIORITY_DEFINITIONS[level].name}). ${factorsStr || 'Standard priority assessment'}`;
  }
}
```

## Calculation Example

### Sample Item: Production Database Outage

```
Item: "CRITICAL: Production database unreachable, all users affected"

Severity Analysis:
- User Impact: 10/10 (All users, complete outage)
- Data Risk: 8/10 (Potential data loss during outage)
- Revenue Impact: 10/10 (Complete service disruption)
- Security Risk: 5/10 (No direct security issue)
- Reputation Risk: 9/10 (Public-facing outage affects trust)

Priority Score:
= (10 × 0.25) + (8 × 0.25) + (10 × 0.20) + (5 × 0.20) + (9 × 0.10)
= 2.5 + 2.0 + 2.0 + 1.0 + 0.9
= 8.4 × 10 = 84

Priority Level: P1 (score 84, just below P0 threshold)

Note: Due to "CRITICAL" keyword override, this would be classified as P0

Recommended Response: IMMEDIATE: Page on-call, assemble incident response team
SLA Deadline: 15 minutes from detection
Escalation Path: On-call → Team Lead → Engineering Manager → VP Engineering
```

## Best Practices

### Do's

1. **Define clear criteria** - Team should agree on what constitutes each level
2. **Respect the SLAs** - Response times must be achievable and enforced
3. **Escalate appropriately** - P0/P1 require immediate escalation
4. **Document everything** - Especially for P0/P1 incidents
5. **Review and adjust** - Regularly review priority assignments

### Don'ts

1. **Don't overuse P0** - Everything P0 = nothing P0 (cry wolf effect)
2. **Don't ignore P3/P4** - Technical debt accumulates
3. **Don't skip post-mortems** - Learn from every P0/P1
4. **Don't game the system** - Honest classification helps everyone

## Common Pitfalls

| Pitfall | Impact | Mitigation |
|---------|--------|------------|
| P0 inflation | Alert fatigue, ignored pages | Enforce strict P0 criteria |
| Unclear criteria | Inconsistent classification | Document decision matrix |
| No follow-up | Same incidents recur | Require post-mortems |
| Ignoring P3/P4 | Technical debt accumulates | Allocate % capacity to lower priorities |
